# 非顺从模型下的大型 App 收敛方案(project-9 取证)

> 8 个子代理:取证确认 → 3 个不依赖模型顺从的方案 → 对抗评审 → 综合。

## 底线

A fully-working large app is NOT reliably achievable with MiniMax-M3, and no in-loop prompting will make it so — the model provably re-emits byte-identical output after being shown the exact authoritative API. BUT two facts from the project-9 log change the strategy decisively: (1) the single largest failure cluster (68 of 102 arg-mismatches — every ChartConfig.apply rejection) is a GUARD FALSE POSITIVE, not a model error: the model correctly declared apply(Context, Chart<?>) once and called it with real MPAndroidChart subtypes PieChart/BarChart/LineChart/HorizontalBarChart; this code compiles under javac, but SymbolTable.isAssignable returns false because superClassByClass has no entry for library types (AndroidSourceGuard.java:598 + SymbolTable.java:233). (2) The genuine model failures (45 missing-method rejections: RecordDao.summarizeByCategory x10, TagDao.softDelete x5, TemplateDao.findById x3, RecordRepository.CategorySummary x3) are deterministically auto-stubbable. So the honest answer: fix the guard's library-subtype blindness (turns ~half the failures into passes with zero model involvement), deterministically stub the genuinely-invented members at merge (turns most of the rest into a buildable APK with marked TODO bodies), and add a same-defect-set fast-abort so an irreducible contradiction ends in seconds with a clear report instead of a 20-minute death loop. Result: a buildable/installable APK with enumerated stub debt in the common case, a fast clear diagnosis in the worst case — never the silent 20-min loop. \"Feature-perfect with no stubs\" is the one thing this model cannot guarantee.

## 选定方案

A four-layer deterministic stack that never depends on the model obeying context, ordered cheapest-and-most-impactful first. LAYER A (guard precision — the biggest single win, fixes 68 rejections, no model involvement): in AndroidSourceGuard.validateCustomMethodCalls before line 598, skip the arg-mismatch verdict for any parameter position whose DECLARED type is a non-generated/library type that the SymbolTable cannot resolve (mirror the existing receiver-side inheritsExternalApi deferral at line 591, but for arguments). Concretely: when hasMethodSignature fails, only emit the violation if EVERY declared candidate signature's params are all guard-resolvable types; if any candidate param is an unseen library type (e.g. Chart) and the arg is a class the guard doesn't own, defer to javac. This is zero-bypass: it only stops asserting a mismatch the guard cannot actually prove, exactly as the receiver path already does. LAYER B (parser hygiene — prerequisite, makes the digest/hint/symbol-table authoritative-clean): add the new-expression guard to all THREE polluting method loops — AndroidSourceGuard.java:388 (add `if (isNewExpressionReference(content, methodMatcher.start(1))) continue;` after line 392), JavaApiReconciler.java:130, and JavaApiDigest.java:126 (the Stage-1 digest the model literally SEES). Empirically verified: this skips IllegalArgumentException/ContentValues/Record (from throw new / = new / return new) and keeps real methods countByCategory and apply. LAYER C (deterministic stub coercion — converts genuine model-invented members into a buildable tree): a new StubReconciler.reconcile(tempDir) sibling after JavaApiReconciler.reconcile at FileOperationsWriter.java:47, before sourceGuard.validate at :50. It builds the (now clean) SymbolTable over the assembled temp tree, re-derives the exact missing-method set the guard would reject (members on NO class, after typo-repair already ran), and for each genuinely-absent member writes a real compiling method/overload/ctor onto the callee with inferred param types, conservative return type, body throwing UnsupportedOperationException(\"STUB\"), tagged // ANDROIDBUILDER-STUB and recorded in a StubLedger. The unchanged guard then validates the self-consistent tree. This makes the cross-file design gap (summarizeByCategory invented by 2 callers) collapse to one frozen signature the SYSTEM owns, not the model. LAYER D (fast-abort fuse — kills the 20-min death loop): in the retry loop at AgentService.java:1205, after rememberFailure, if the NORMALIZED defect set is byte-identical across 2 consecutive attempts (the log shows attempts 4→5 identical at lines 75-78 vs 101-104), abort immediately with a deterministic ContractConflictReport instead of draining all 5 attempts. Reuse FailureFingerprintPolicy.normalizeMessage (make it package-visible, currently private at line 130) for the comparison. This survives total non-compliance because it triggers ON non-compliance.

## 分阶段


### Stage 0 (风险 low) — Ship the parser fix and guard-precision fix — pure deterministic wins, no new architecture, fixes the largest rejection cluster (68 false positives) plus polluted hints

单独价值: Yes — independently converts ~67% of the project-9 arg-mismatch failures (the ChartConfig cluster) into passes and removes the phantom-method pollution from every hint/digest, with zero new components and zero model dependency. This is the highest-leverage change in the whole plan.

改动:
- AndroidSourceGuard.java:388 method loop: add `if (isNewExpressionReference(content, methodMatcher.start(1))) { continue; }` immediately after computing owner (after line 392). Empirically verified via compiled-regex probe: skips IllegalArgumentException/ContentValues/Record (throw new / = new / return new), keeps countByCategory and apply.
- JavaApiReconciler.java:130 method loop and JavaApiDigest.java:126 method loop: add the equivalent new-expression skip (back-walk to check the token before the captured name is not 'new'/'return'/'throw'). JavaApiDigest is the Stage-1 non-truncatable digest the MODEL sees, so cleaning it directly de-pollutes the authoritative context.
- AndroidSourceGuard.java:598 (validateCustomMethodCalls): before emitting 'method argument mismatch', add an argument-side library-type deferral mirroring inheritsExternalApi at line 591 — if any candidate signature has a declared param of a type the SymbolTable cannot resolve (not hasClass, not a known primitive/boxed/Context), and the corresponding arg is a class the guard does not own, defer to javac instead of rejecting. This alone turns all 68 ChartConfig.apply(Context, Pie/Bar/Line/HorizontalBar) rejections into passes on code that already compiles.
- Add SymbolTableTest/AndroidSourceGuardTest cases: the 3 parser pollutants no longer register as methods; apply(Context,Chart<?>) called with PieChart passes; a real arity mismatch still rejects; a characterization test that the existing AndroidSourceGuardTest stays green.

### Stage 1 (风险 low) — Turn the 20-minute death loop into a seconds-long deterministic outcome — the stated core value, independent of any app actually building

单独价值: Yes — this delivers the explicitly-stated real value ('a 20-minute death loop that ends in failure is itself a failure; converting it to a fast, deterministic, clear outcome is real value') with no dependency on Stage 2 and no model compliance. It triggers precisely because the model is non-compliant.

改动:
- FailureFingerprintPolicy.java:130: change normalizeMessage from private to package-visible so the loop can compare normalized defect sets.
- AgentService.java:1205 retry loop: after rememberFailure (line 1969 path), compute the normalized defect SET for this attempt; if it is byte-identical to the previous attempt's set (the log proves attempts 4→5 are identical), break out and fail fast rather than running attempts to POLICY_REWRITE_ATTEMPTS=5. The repeatedRetryContext machinery at line 55 already detects repeats but only injects more context — redirect that signal to abort.
- New ContractConflictReport renderer: from the repeated defect set, produce a human-readable diagnosis grouping multi-call-shape and missing-member defects (e.g. 'RecordDao.summarizeByCategory(long,long,String) called from StatisticsService.java and RecordRepository.java is never declared; ChartConfig.apply is declared once as apply(Context,Chart) — N call-sites'). Surface it to the user at the AgentService.java:1545 termination path.
- Tests: identical defect set across 2 attempts aborts before attempt 3; a set that genuinely shrinks between attempts does NOT abort (progress is preserved).

### Stage 2 (风险 medium) — Convert genuinely-invented members (the 45 missing-method rejections) into a buildable APK with tracked stub debt

单独价值: Partially — produces a buildable/installable APK whose stubbed code paths return defaults/throw until filled. 'Working' here means 'builds, installs, launches, with a greppable enumerated TODO list', NOT feature-complete. This is a genuine semantic compromise the user must accept; gate it behind a flag so it can be disabled to get only the fast-abort (Stage 1) behavior for invented members.

改动:
- New StubReconciler.java: reconcile(File tempDir). Build the (Stage-0-clean) SymbolTable over the temp tree via the existing collectSymbolTableForTest seam (AndroidSourceGuard.java:61); re-derive the missing-method/ctor/field set the guard would reject; for each member that exists on NO class (after JavaApiReconciler typo-repair already ran), splice a real compiling member onto the callee .java using the DatabaseContractNormalizer.insertColumnConstants splice technique (DatabaseContractNormalizer.java:95-111). Body throws UnsupportedOperationException for non-void, returns default for primitives; tag // ANDROIDBUILDER-STUB.
- FileOperationsWriter.java:47: insert StubReconciler.reconcile(tempDir) AFTER JavaApiReconciler.reconcile and BEFORE sourceGuard.validate at :50. Order matters: typos repaired first (reuse an existing method), only genuinely-absent names get stubbed; the unchanged guard at :50 still validates the result — a bad stub is caught, never bypassed.
- New StubLedger.java persisted beside ProjectPlanRecord (pattern at AgentService.java:307): record every {class, member, signature, callers}. Surface to user; re-inject as the next round's 'fill these stub bodies' instruction so semantic debt is enumerated, never silent.
- Return-type inference must be conservative: only stub when no declared signature satisfies the call; if two callers infer incompatible return types for the same member, do NOT widen to Object — route to the Stage-1 fast-abort as an irreducible conflict. Tests: summarizeByCategory invented by 2 callers -> 1 stub method, guard passes, tree compiles; incompatible-return conflict aborts.

### Stage 3 (风险 high) — OPTIONAL/long-term: AppContract seed+reconcile (the pending #24/#25 work) as a retry-efficiency optimization — NOT a convergence guarantee

单独价值: Marginally — catches missing-callee and resource-ownership impossibilities a few seconds earlier across all edges at once. Real but secondary leverage; the verdicts correctly show the 'collapsed by construction' claim is undermined by the same library-blindness that Stage 0 fixes at the guard, so this is not the convergence mechanism it was pitched as.

改动:
- Implement only the closed-world, compliance-independent checks: F1 callee-class existence and F4 resource single-owner across all edges at plan time (these are existence/ownership, not type-shape, so they cannot false-positive on library subtypes the way F2 does).
- Explicitly DO NOT make F2 argument-matching load-bearing — it has the same library-supertype blindness as the guard (it would re-flag apply(Context,PieChart) against apply(Context,Chart)); treat F2 as best-effort only, and seed a small static catalog of known library supertypes (Chart -> Pie/Bar/Line/HorizontalBarChart) if F2 is used at all.
- Only realized=true body-derived signatures are authoritative for consumers; the seed is advisory — never freeze a pre-plan guess as truth.

## 立即可做的确定性小修

- Parser fix (AndroidSourceGuard.java:388): add `if (isNewExpressionReference(content, methodMatcher.start(1))) { continue; }` after line 392 — empirically verified to drop the IllegalArgumentException/ContentValues/Record phantoms while keeping real methods. Mirror in JavaApiReconciler.java:130 and JavaApiDigest.java:126 (the latter is the digest the model sees). This is the confirmed method-parser bug.
- Guard library-subtype fix (AndroidSourceGuard.java:598): defer the arg-mismatch verdict when a declared param is an unseen library type and the arg is a class the guard does not own — mirrors the existing receiver-side inheritsExternalApi deferral at line 591. Single biggest win: eliminates 68 false-positive rejections (all ChartConfig.apply) on code that already compiles under javac.
- Fast-abort fuse (AgentService.java:1205 + FailureFingerprintPolicy.normalizeMessage made package-visible at line 130): if the normalized defect set is byte-identical across 2 attempts, abort with a ContractConflictReport instead of burning all 5 attempts/20 min. Converts the death loop into a seconds-long clear outcome — triggers precisely on the non-compliance the log proves.
- Stop trusting model self-summaries: log line 7006 shows the model asserting 'no signature mismatches remain in this phase' while the guard immediately found mismatches. The deterministic guard/linter must remain the sole consistency authority — never gate on the model's own claim of correctness.

## 诚实的边界

WILL achieve: (1) Eliminate the dominant failure cluster — 68 of 102 arg-mismatches are a guard false-positive on compilable library-subtype calls; the Stage-0 fix makes them pass with zero model involvement. (2) Turn the genuinely-invented-member failures (45 missing-method rejections) into a buildable APK via deterministic stubs, with every stub greppable and ledgered. (3) Kill the 20-minute death loop — same-defect-set abort yields a seconds-long, precise diagnosis. WILL NOT achieve: a feature-complete app with zero stubs from this model. The evidence is conclusive that context-based convergence is exhausted — at log line 7006 the model declared its own work consistent while it was not, and after being shown the exact one-method truth it re-emitted byte-identical output (attempts 4→5). The stubbed code paths (e.g. summarizeByCategory aggregation) compile and run but return defaults until a human or a later targeted pass fills them, so 'working' means 'builds, installs, launches, with an enumerated TODO list', not 'every feature functions'. The user's stated 'no compromise' is in direct tension with this: a stub body IS a behavioral compromise. The honest framing: a buildable APK with bounded, surfaced, greppable stub debt is a vastly better outcome than a silent 20-minute loop that ships nothing — but if the user truly requires zero stubs, the realistic deliverable from this model is the fast-abort + guard-precision fixes (a clean fail-fast and far fewer false rejections), not a guaranteed complete app. One caveat to verify in implementation: a subset of the non-Chart arg-mismatches (sumAmount, computeRange) and the missing-method members are genuine model errors that the guard-precision fix will NOT mask — those correctly remain rejections and route to stub-or-abort, which is the intended behavior.
